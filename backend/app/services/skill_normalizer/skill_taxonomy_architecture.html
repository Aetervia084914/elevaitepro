🏗 Architecture Overview — Skill Detection Taxonomy

Goal: Build the world-class skill detection taxonomy in PostgreSQL by fusing Lightcast 9-industry skill maps, ESCO v1.2 (13,890 skills), ONET 29.1 (35 skill elements x 1,018 SOC), BLS wage + growth signals, CEDEFOP EU demand forecasts, Kaggle job-posting empirical validation, and Wikidata altLabels. Expose via FastAPI as a flat skills[] list from full resume raw text with no section detection.

Industry Scope: 9 industries only. All taxonomy entries, domain hints, phrase patterns, confidence scoring, and LLM grounding are scoped to these 9 Lightcast industries and their domain_hint slugs:

SOC codes outside these 9 categories (e.g., healthcare 29-x, legal 23-x) are mapped to the nearest Lightcast industry: healthcare → analysis, legal → business, education → business, social services → hr. No “general” or “other” domain is used.

Language Scope: English (US), English (UK), English (India) only. Non-English documents are detected at Stage 2 and rejected with a 400 UNSUPPORTED_LANGUAGE error. ESCO altLabels are filtered to English-language entries only during Stage 0B ETL. No multilingual alias lookup, locale-routing, or translated prompt injection is performed.

Two processing tracks: (1) Offline ETL — build the PostgreSQL taxonomy once from all datasets (Stage 0). (2) Online inference — per-request file-to-text conversion + multi-layer taxonomy matching + LLM fallback (Stages 1–6). Every stage writes a row to stage_result.

Data structure note (validated): skill_map_*.json files are ID index files only (contain lightcast_skill_id + URL, no names). Canonical skill data lives in Skills_<industry>/<skill_id>.json files — 18,830 skill files + 27,057 occupation files across all 9 industries, all present on disk. Business index map issue is resolved — Skills_Business/ has 1,553 individual skill files loaded directly.

Stage 2 — Document → Raw Text Extraction (No Section Detection)
Design Decision
Full raw text only — no section detection. The entire document is extracted as a single flat string. Skills are detected wherever they appear: headers, bullets, prose, footnotes. This is intentional to avoid missing skills in non-standard resume layouts.
Format Handlers
.docx → python-docx (paragraphs + tables) · .doc → docx2txt + LibreOffice subprocess fallback · .pdf → pdfminer.six (text-layer primary) + pymupdf OCR (scanned fallback) · .odt / .ods → odfpy · .txt → UTF-8 decode, BOM strip
Text Cleaning
ftfy (Unicode artefact fix) · regex (whitespace normalisation, bullet char strip, control char removal) · langdetect (language ID → if detected language is NOT en, reject immediately with HTTP 400 UNSUPPORTED_LANGUAGE; EN(US)/EN(UK)/EN(IN) all detected as 'en')
Text Size Cap
MAX_TEXT_CHARS = 50,000 (covers ~20 pages). If extracted text exceeds 50K chars, truncate to first 50K with a logged warning. Store original_char_count in stage_result.payload for audit. Prevents 4-gram explosion (~800K n-gram candidates) on 100-page PDFs. Implemented as a post-extraction guard before yielding text to Stage 3.
Dataset Files
None at this stage. Language detection result used only as an English gate — non-English documents are rejected here and never reach Stage 3.

Phase 2 — Multi-Layer Skill Detection Against PostgreSQL Taxonomy
Stage 3 — Taxonomy-Driven Skill Matching (3 sub-stages)
3A — Exact Match + Alias Resolution (Primary Extraction)
Purpose
Tokenise raw text into 1–4-gram windows. Min token guard: skip any token with len(token) < 2. For 2–3 char tokens (e.g. Go, R, C#), require exact case-sensitive match against taxonomy_aliases.alias_lower — no trigram fuzzy for short tokens. Query taxonomy_aliases table for exact canonical name match and alias hits. Aliases include: Lightcast skill aliases, ESCO altLabels, ONET technology tool names, Wikidata altLabels. Highest confidence path — weight 1.0 (exact) / 0.92 (alias).
DB Query
SELECT ta.skill_id, ts.canonical_name, ta.confidence_modifier FROM taxonomy_aliases ta JOIN taxonomy_skills ts USING (skill_id) WHERE ta.alias_lower = ANY($1) — batch n-gram lookup. canonical_name lives on taxonomy_skills, not on taxonomy_aliases; JOIN required. confidence_modifier is the correct column name (NUMERIC(4,2) default 0.0). Redis L1 cache: alias:{md5(ngram)} TTL 3600 for hot terms.
Why PostgreSQL
All dataset aliases (ESCO altLabels + Wikidata + ONET tools = 100K+ entries) are pre-indexed with GIN trigram index on alias_lower. Sub-5ms p99 lookup on 100K alias table. Zero filesystem I/O at request time.

Purpose
Catch skills expressed as verb phrases in prose rather than listed keywords. E.g. “built and deployed REST APIs” → api_development (0.70). “led cross-functional teams” → stakeholder_management (0.70). 3C — Semantic Embedding Similarity (Tertiary Extraction)
Trigger / Input
Runs on raw text windows that were NOT matched by 3A or 3B. After 3A, matched character span offsets are recorded in a matched_spans: set[tuple[int,int]] in-memory set. After 3B, phrase-matched spans are added. 3C receives only the unmatched sliding windows (4-gram context blocks of ~120 chars) — never re-processes spans already claimed with confidence ≥ 0.65. This avoids overwriting 3A exact matches (confidence 0.92).
Purpose
Catches paraphrased skills and domain-specific jargon absent from alias tables. Encodes each unmatched context window to a 384-dim vector and compares against pre-indexed skill embeddings in taxonomy_skill_embeddings using pgvector cosine similarity. Threshold: similarity ≥ 0.82 (distance ≤ 0.18). Assigns base confidence 0.65 for semantic hits. Capped at 10 semantic candidates to prevent noise from displacing higher-confidence exact/alias matches in the 30-skill cap.
Embedding Model
sentence-transformers/all-MiniLM-L6-v2 — 384-dim, ~2 ms/sentence on CPU. Loaded once at process startup via model_singleton.lifespan (FastAPI lifespan context manager); stored as app.state.encoder. Injected per-request via Depends(get_encoder) — zero model-load overhead at request time. Skill embeddings pre-computed at taxonomy build time via model_singleton.encode_texts(), stored in taxonomy_skill_embeddings.embedding vector(384).
DB Query
SELECT tse.skill_id, ts.canonical_name, tse.embedding <=> $1 AS dist FROM taxonomy_skill_embeddings tse JOIN taxonomy_skills ts USING (skill_id) WHERE tse.embedding <=> $1 <= 0.18 ORDER BY dist LIMIT 5 — canonical_name is on taxonomy_skills (not on taxonomy_skill_embeddings), JOIN required. <=> = pgvector cosine distance; threshold 0.18 = similarity ≥ 0.82. IVFFlat ANN index accelerates the ORDER BY dist scan.

Stage 4 — Confidence Scoring, Deduplication & Demand Signal Boost
Purpose
Merge candidates from 3A+3B+3C. Deduplicate by skill_id (keep highest base_weight per skill). Then apply in order:
Step 1 — Hard blacklist exclusion (pre-score): DELETE candidates WHERE skill_id IN (SELECT skill_id FROM taxonomy_skill_blacklist WHERE whitelist_override = false). Hard removal — no penalty term; a blacklisted skill is never scored.
Step 2 — Composite confidence score:
score = min(base_weight + onet_importance_bonus + demand_growth_bonus + cedefop_shortage_bonus - low_freq_pattern_penalty, 1.0)
Base weights by source: exact match = 1.0 · alias hit = 0.92 · phrase pattern = 0.70 · semantic = 0.65. Score clamped to 1.0 maximum.
Step 3 — Floor filter: read taxonomy_skills.confidence_floor (NUMERIC(4,2) default 0.62); remove any candidate where score < confidence_floor.
Step 4: Sort descending by score. Cap at 30 skills.
Signals Used
ONET importance score (from 0B → taxonomy_skills.onet_importance) — bonus +0.05 if importance ≥ 4.0 in matched SOC · BLS growth rate (from 0C → taxonomy_skill_signals.bls_growth_rate) — bonus +0.03 if occupation 10-yr growth > 10% · CEDEFOP shortage index (from 0C) — bonus +0.02 for shortage skills · Kaggle frequency (from 0C) — validates pattern; low-freq patterns get penalty -0.05
DB Query
Blacklist pre-filter: SELECT skill_id FROM taxonomy_skill_blacklist WHERE whitelist_override = false — loaded once at startup into an in-memory set.
Score components + floor: SELECT ts.skill_id, ts.onet_importance, ts.confidence_floor, tss.bls_growth_rate, tss.cedefop_demand_score, tss.shortage_flag FROM taxonomy_skills ts LEFT JOIN taxonomy_skill_signals tss USING (skill_id) WHERE ts.skill_id = ANY($1).
confidence_floor is on taxonomy_skills (not taxonomy_skill_signals). LEFT JOIN used so skills without signal rows still score from base_weight.

Phase 5 — Profile Inference (Industry, Occupation & Job Level)
Stage 7 — Industry, Occupation Group & Job Level Profiling
Purpose
Given the final extracted skill_ids, infers: (1) top matching industries, (2) top matching occupation groups, and (3) a seniority job-level band — all with confidence scores. Runs after Stage 6, adds profile fields to the HTTP response.
Industry Logic
Weighted-vote across taxonomy_skill_industry per career_area. Vote weight = skill_confidence × industry.weight. Confidence = share of total weighted votes. Returns top 5. Title-based industry detection (first 500 chars) applies 5× boost + 0.5× dampening on non-title top industry. 14 industries supported: IT, Engineering, Finance, Business & Management, Marketing & Sales, Design & Creative, HR, Data & Analytics, Customer Service, Healthcare, Manufacturing, Education, Logistics & Supply Chain, Defence & Security. QA/Test roles detected as IT via title patterns.
Occupation Logic
PRIMARY: Same weighted-vote approach grouped by occupation_group column in taxonomy_skill_industry (256 distinct groups). ENRICHMENT: skill_occupations → occupations provides nested per-occupation detail (name, ISCO code, skill overlap) within each top group. Returns top 5 occupation groups with up to 5 nested occupations each.
Job Level Logic
Three prioritised signals fused by max-rank: (1) Seniority tokens in matched occupation_group names — narrow sets (“architect”, “principal”, “tech lead”, “senior”, “director” etc.) so broad groups like “Project Management” do not falsely elevate level; (2) ONET importance — only activated when ≥30 % of final skills have a non-zero value (column is sparsely populated); bands entry<2.2→mid<3.4→senior<4.2→lead; (3) Distinct career-area count as experience-breadth proxy (1=entry, 2–4=mid, ≥5=senior). Spec-ratio deliberately excluded — Lightcast tags every skill as “Specialized Skill”, making it a non-discriminating constant. Confidence = f(signal agreement, ONET coverage).

▸ Group 8 — Kaggle Job Postings (Empirical Validation)
DB TABLE · Kaggle job_postings_raw & kaggle_job_postings — postgresql+psycopg://admin:admin@localhost:5432/elevaite
Tables
job_postings_raw — raw job posting text · kaggle_job_postings — structured job posting records (title, company, description, skills_mentioned)
Why Critical
Real-world empirical validation. Each phrase pattern in taxonomy_phrase_patterns is validated by computing how often it co-occurs with explicit skill keywords in actual job postings. Patterns that never appear in real postings are flagged as low-confidence. This prevents false-positive phrase detections — the #1 failure mode in rule-based skill extraction.
Usage
SQL: SELECT pattern, COUNT(*) co_occurrence FROM job_postings_raw WHERE description ~* pattern GROUP BY pattern. Result stored as taxonomy_phrase_patterns.kaggle_frequency. Also used to mine new phrase patterns not yet in ONET task statements.
Stage Used
