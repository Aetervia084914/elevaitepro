STAGE 1 FILE I/O REDIS File Upload, Format Detection & Cache Check

Logic Steps

  • 1a. Size validation: reject if 0 bytes or > 10 MB (413 error)
  • 1b. Format detection: python-magic detects MIME type from first 8 KB. Falls back to file extension mapping if magic fails
  • 1c. Content hash: SHA-256(file_bytes) → 64-char hex digest stored as ctx.content_hash
  • 1d. Redis cache lookup: GET skills:{content_hash} — if hit, loads ctx.final_skills[] and short-circuits to cache-hit path

Redis Key Details

Key: skills:{content_hash}
Value: JSON array of canonical skill names — e.g. ["Python","SQL","Docker"]
TTL: 86 400 seconds (24 hours, from CACHE_TTL env var)
Serializer: orjson.dumps / orjson.loads

Output

  • Creates PipelineContext with request_id (UUID4), content_hash, file_format, file_size_bytes, cache_hit
CACHE HIT Short-Circuit Path (skip Stages 3–6)

Logic

  • Run Stage 2 text extraction anyway (non-fatal) so ctx.raw_text is available for Stage 7 title-based industry detection
  • If ctx.final_skill_ids is empty, fetch from DB: SELECT skill_ids FROM skill_detections WHERE content_hash = %s
  • Build synthetic SkillCandidate[] from cached skill names + DB skill IDs (confidence = 0.9, source = "cache")
  • Jump directly to Stage 7 for profile enrichment → persist profile → return response

STAGE 2 FILE I/O Document → Raw Text Extraction & Language Gate

Extraction Handlers

  • .docx → python-docx (paragraphs + table cells)
  • .doc → docx2txt first, fallback to LibreOffice soffice --headless
  • .pdf → pdfminer (text layer), fallback to PyMuPDF/fitz
  • .odt/.ods → odfpy
  • .txt → UTF-8 decode with BOM strip

Post-Processing

  • Clean text: ftfy.fix_text() → strip control chars → normalize bullets → collapse whitespace
  • Truncate: cap at 50 000 chars
  • Language gate: langdetect on first 2 000 chars — rejects non-English with HTTP 400

Output

  • ctx.raw_text — cleaned text for all downstream stages
  • ctx.detected_language — must be "en"
STAGE 3A PostgreSQL Exact Match & Alias Resolution (Primary Extraction)

Algorithm

  • Tokenise: regex [A-Za-z0-9#\+\.]+ with hyphen/slash joining
  • N-grams: build 1-to-4-gram windows from tokens (skip single-char tokens)
  • Deduplicate: keep first (start, end, original) per lowered n-gram
  • Batch DB lookup: n-grams batched in 2 000-row chunks against taxonomy_aliases
  • Confidence: exact match → 1.0, alias match → 0.92 + confidence_modifier from alias row
  • Short aliases ≤3 chars filtered unless exact match or literally found in text

Tables Read

taxonomy_aliases Maps lowered alias strings → skill_id. Columns: alias_lower, skill_id, confidence_modifier
taxonomy_skills JOIN to retrieve canonical_name for each matched skill_id

SQL Query

SELECT ta.alias_lower,
       ta.skill_id,
       ts.canonical_name,
       ta.confidence_modifier
FROM taxonomy_aliases ta
JOIN taxonomy_skills ts USING (skill_id)
WHERE ta.alias_lower = ANY(%s)

Output

  • ctx.candidates[] — SkillCandidate with source_stage="3A"
  • ctx.matched_spans[] — (start, end) positions consumed by 3A
STAGE 3B PostgreSQL Phrase Pattern Matching (Secondary Extraction)

Algorithm

  • Patterns are pre-loaded at startup from taxonomy_phrase_patterns → stored on app.state.phrase_patterns
  • Each pattern is a compiled regex with skill_id, base_confidence, kaggle_frequency, domain_hint, priority
  • For each pattern: search ctx.raw_text → on match, compute confidence
  • Low-frequency penalty: −0.05 if kaggle_frequency < 50
  • Co-occurrence boost: +0.03 if matched skill co-occurs with existing candidates (via taxonomy_skill_cooccurrence)
  • Look up canonical_name from taxonomy_skills for each matched skill_id

Tables Read

taxonomy_phrase_patterns Regex patterns → skill_id. Pre-loaded at startup. Cols: pattern_id, pattern_regex, skill_id, base_confidence, kaggle_frequency, domain_hint, priority
taxonomy_skill_cooccurrence Pairwise skill co-occurrence counts. Queried per candidate to boost confidence if co-occurring skills already detected
taxonomy_skills Canonical name lookup for each matched skill_id

Output

  • Appends new SkillCandidates (source="3B") or upgrades confidence of existing ones
STAGE 3C PostgreSQL + pgvector ML MODEL Semantic Embedding Similarity (Tertiary Extraction)

Algorithm

  • Build unmatched windows: 120-char sliding windows (step 60) over text regions NOT covered by 3A/3B matched_spans. Skip windows with >50% overlap. Max 200 windows
  • Encode windows: sentence-transformers/all-MiniLM-L6-v2 (384-dim) — batch_size=32, normalize_embeddings=True
  • pgvector nearest-neighbour: for each window vector, query taxonomy_skill_embeddings using cosine distance operator <=>
  • Threshold: cosine distance ≤ 0.18 (i.e. similarity ≥ 0.82)
  • Cap: max 10 semantic candidates total (MAX_SEMANTIC_CANDIDATES)
  • Base confidence: 0.65 for all semantic matches

Tables Read

taxonomy_skill_embeddings Pre-computed 384-dim vectors for each skill. Column: embedding vector(384). Uses pgvector <=> cosine distance operator
taxonomy_skills JOIN to retrieve canonical_name for matched skill_id

SQL Query

SELECT tse.skill_id,
       ts.canonical_name,
       tse.embedding <=> %s::vector AS dist
FROM taxonomy_skill_embeddings tse
JOIN taxonomy_skills ts USING (skill_id)
WHERE tse.embedding <=> %s::vector <= 0.18
ORDER BY dist LIMIT 3  -- per window
Model: sentence-transformers/all-MiniLM-L6-v2
Dim: 384  |  Loaded at: startup via init_encoder(app) → stored on app.state.encoder
STAGE 4 PostgreSQL Confidence Scoring, Dedup & Demand Signal Boost

Filtering Pipeline

  • Step 1 — Blacklist: remove candidates whose canonical_name appears in blacklist (pre-loaded from taxonomy_skill_blacklist)
  • Step 1b — Dedup: keep highest-confidence candidate per skill_id
  • Step 2 — Fetch signals: batch-query taxonomy_skills + taxonomy_skill_signals for ONET importance, BLS growth, CEDEFOP demand, skill_type, category_flags
  • Step 2b — Type filter: remove "Common Skill", "skill_group", and "transversal" flagged candidates
  • Step 3 — Demand bonus: +0.05 if ONET importance ≥ 4.0, +0.03 if BLS growth > 10%, +0.02 if shortage_flag
  • Step 4 — Floor filter: remove candidates below per-skill confidence_floor (default 0.62)
  • Step 5 — Sort & cap: sort by confidence descending, keep top 30 (MAX_SKILLS)

Tables Read

taxonomy_skills onet_importance, confidence_floor, skill_type, skill_category_flags
taxonomy_skill_signals bls_growth_rate, cedefop_demand_score, shortage_flag
taxonomy_skill_blacklist Pre-loaded at startup. Cols: term_lower, whitelist_override. Filters noise terms

SQL Query (signals)

SELECT ts.skill_id,
       ts.onet_importance,
       ts.confidence_floor,
       tss.bls_growth_rate,
       tss.cedefop_demand_score,
       tss.shortage_flag,
       ts.skill_type,
       ts.skill_category_flags
FROM taxonomy_skills ts
LEFT JOIN taxonomy_skill_signals tss
  USING (skill_id)
WHERE ts.skill_id = ANY(%s)

Output

  • ctx.final_skills[] — canonical names
  • ctx.final_skill_ids[] — integer skill IDs
  • ctx.skill_count — 0–30
STAGE 5 LLM API REDIS PostgreSQL LLM Fallback (Conditional: < 3 skills detected)

Trigger Condition

  • Only if ctx.skill_count < 3 — otherwise this stage is SKIPPED

Algorithm

  • LLM cache check: GET llm_skills:{content_hash} — avoids re-calling the LLM for the same document
  • LLM call: POST to {LLM_BASE_URL}{LLM_ENDPOINT} with system+user prompt
  • Prompt injects top-500 canonical skills (pre-loaded at startup from taxonomy_skills ORDER BY onet_importance DESC) as grounding
  • LLM returns {"skills": ["name1", "name2", ...]}
  • Cache result: SETEX llm_skills:{content_hash} 86400 [json]
  • Validate: look up LLM-returned names against taxonomy_skills to get valid skill_ids
  • Merge new LLM candidates (confidence 0.75, source="5_llm") into existing set, re-run Stage 4 scoring

LLM Configuration

Endpoint: http://localhost:8005/openchat (configurable via LLM_BASE_URL + LLM_ENDPOINT)
Model: gpt-5.4 (configurable via LLM_MODEL)
Auth: Bearer {OPENAI_API_KEY}
Max tokens: 400  |  Temperature: 0.0
HTTP client: httpx.AsyncClient (timeout 30s)

Redis Keys

Key: llm_skills:{content_hash}
Value: JSON array of LLM-extracted skill names
TTL: 86 400 seconds (24 hours)

Tables Read

taxonomy_skills Validate LLM names: WHERE lower(canonical_name) = ANY(%s)
STAGE 6 DB WRITE REDIS WRITE Validation, Persistence & Cache

Logic

  • Validation: strip candidates with null skill_id or empty canonical_name
  • Persist skill_detections: INSERT/upsert final skills, skill_ids, extraction_method, skill_count keyed by request_id
  • Persist trace: one row per candidate in skill_detection_trace with accepted/rejected status and confidence
  • Cache to Redis: SET skills:{content_hash} [json] EX 86400

Tables Written

skill_detections Final output row. Cols: request_id (PK), content_hash, file_format, detected_language, skills[], skill_ids[], extraction_method, skill_count, completed_at. ON CONFLICT (request_id) DO UPDATE
skill_detection_trace One row per candidate. Cols: request_id, skill_id, source_stage, matched_text, confidence, accepted (bool), reject_reason
STAGE 7 PostgreSQL (5 tables) CSV FILE DB WRITE Industry, Occupation Group & Job Level Profiling

Step 1 — Fetch All Signals

  • Industry rows: taxonomy_skill_industry — industry_name, career_area, occupation_group, weight for each final skill_id
  • ONET signals: taxonomy_skills — onet_importance, skill_type, skill_category_flags
  • Lightcast occupations: JOIN taxonomy_skills → skill_occupations → occupations via lightcast_id
  • Breadth & count: per-occupation skill breadth from skill_occupations, total occupation count from occupations
  • ISCO labels: loaded from data/ESCO/ISCOGroups_en.csv (cached on first call)

Tables Read

taxonomy_skill_industry 33 112 rows. Maps skill_id → industry_name, career_area, occupation_group, weight. PRIMARY signal for both industry + occupation group
taxonomy_skills ONET importance, skill type, category flags for job level inference
skill_occupations Lightcast skill→occupation links. Used for ENRICHMENT (nested occupation detail), not primary grouping
occupations Lightcast occupations (id, name, isco_code). Total count used for IDF calculation

Step 2 — Industry Aggregation

  • Weighted vote: for each skill's industry row: vote = skill_confidence × weight. Sum votes per normalised industry name
  • Industry normalisation: maps raw labels (e.g. "it", "engineering", ISCO group names) to clean display names (14 industries)
  • Title detection: first 500 chars of raw_text matched against 40+ regex patterns → industry hint
  • Title boost: 8× on title-detected industry, 0.3× dampening on current top if it differs, extra 0.4× dampening on IT if title is non-IT
  • Softmax: confidence = score / total_scores → top 5 industries returned

Step 3 — Occupation Aggregation

  • Primary (architecture-aligned): weighted vote across taxonomy_skill_industry.occupation_group
  • Enrichment: skill_occupations → occupations for nested per-occupation detail within each top group
  • Scoring: TF-IDF (IDF = log(N / (1+df))), overlap × 10 + tfidf × (1 + 0.5 × coverage)
  • Softmax: log1p-scaled softmax with temperature=0.8 for per-occupation confidence
  • Top 5 occupation groups, each with up to 5 nested occupations + matching_skills

Step 4 — Job Level Inference

  • Signal 0 (highest priority): title keywords in first 500 chars → regex patterns for lead/senior/mid/entry
  • Signal 1: seniority tokens in occupation group names (architect, principal → lead; senior, expert → senior)
  • Signal 2: ONET importance average. Bands: lead ≥ 4.2, senior ≥ 3.4, mid ≥ 2.2, entry ≥ 0. Requires ≥ 30% non-zero coverage
  • Signal 3: career-area diversity count. ≥ 6 areas → senior, ≥ 3 → mid, else entry
  • Combine: title keyword overrides all; otherwise max rank of remaining signals
  • Confidence: 0.45 + 0.30 × signal_agreement + 0.10 × ONET coverage + 0.10 if title found

Tables Written

skill_profile_detections Stores industries (JSONB), occupation_groups (JSONB), job_level, job_level_confidence, job_level_onet_avg, job_level_evidence (TEXT[]). Keyed by request_id. ON CONFLICT DO NOTHING
FINALIZE compute_best_match() + Persist Audit Trail + Return Response

Best-Match Post-Processing

  • Best industry: highest-confidence IndustryMatch
  • Best occupation: industry-aware scoring across all nested occupations — boost 1.8× overlap + 0.20 conf bonus if occupation group aligns with top industry, penalise 0.65× if not aligned (non-IT)
  • Best job level: directly from ctx.job_level
  • Composed into BestMatch object (single top-1 pick for industry, occupation, level)

Persistence

  • stage_results: all accumulated StageResultRow objects (stages 1–7 + output) written to stage_results table
  • sessions: updated to pipeline_status='completed' with alias_count, completed_at
stage_results Per-stage audit. Cols: session_id (FK), stage_number, stage_name, status, execution_time_ms, stageoutput (JSONB), error_message, extraction_type='skill'
sessions Updated: pipeline_status → 'completed', alias_count, completed_at
